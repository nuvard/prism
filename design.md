# Требования к проекту Attention Scores

Документ задаёт требования к реализации пайплайна сбора attention scores и связанной статистики. При разработке необходимо придерживаться этих требований.

---

## 1. Окружение и зависимости

- **Виртуальное окружение**: использовать только venv в корне проекта (`.venv`). Создание: `python -m venv .venv`. Запуск кода и тестов — только с активированным этим venv или явным путём к интерпретатору (например, `.venv\Scripts\python.exe` на Windows).
- **Зависимости**: управление только через **requirements.txt**. Файл **pyproject.toml не использовать**. В requirements зафиксировать: `transformers`, `torch`, `pydantic`, `pydantic-settings`, `numpy`, `pytest` (и при необходимости `pyyaml`), `matplotlib` (для визуализации), с указанием версий где нужно. Для поддержки NPU — опциональная зависимость **torch_npu** (установка по необходимости для устройств Ascend и т.п.).
- В README и скриптах явно указывать использование интерпретатора из venv.

---

## 2. Структура проекта

- `src/attention_scores/` — основной пакет (с `__init__.py`).
- `src/visualization/` — пакет визуализации: построение графиков по готовым выводам пайплайна (без вызова модели).
- `configs/` — конфигурации (например, `default.yaml` или `default.json`).
- `tests/` — все тесты; при необходимости подкаталоги с `__init__.py`.
- Корень: `README.md`, `requirements.txt`, `design.md`. Выход пайплайна — в каталог, заданный конфигом (`output_dir`).

## 3. Типизация и расширяемость

- **Все публичные функции и методы** должны иметь полную типизацию: аннотированы все аргументы и возвращаемый тип.
- Использовать `typing` и при необходимости `typing_extensions`. Для структур данных — `TypedDict`, `dataclass` или pydantic-модели с явными типами полей.
- Сигнатуры должны быть спроектированы так, чтобы функционал можно было расширять (добавление новых метрик, полей в вывод) без ломания существующих вызовов. Избегать «магических» позиционных аргументов; предпочитать именованные и структурированные типы.

---

## 4. Конфигурация

- Все параметры пайплайна задаются через конфиг (YAML или JSON). Допускается переопределение через переменные окружения или CLI.
- Обязательные/ожидаемые ключи конфига:
  - `dataset_path` — путь к JSON-датасету;
  - `model_path` — путь к модели (Hugging Face или локальный);
  - `batch_size`, `max_output_len`;
  - `save_every_n_steps`, `save_when_new_important_above_k`;
  - `importance_threshold` (по умолчанию 0.95);
  - `thinking_markers` — список строк для детекции маркеров размышлений;
  - `output_dir` — корневая папка вывода;
  - `save_prefill_attention` — флаг сохранения скоров prefill отдельно;
  - `sparsity_threshold` (опционально) — порог для метрики разреженности;
  - **`device`** — устройство для вычислений: `"cpu"`, `"cuda"` (GPU NVIDIA), `"npu"` (Ascend и др. — через API **torch_npu**) или с индексом (`"cuda:0"`, `"npu:0"`). По умолчанию — авто (cuda при наличии, иначе cpu). Модель и тензоры переносить на выбранное устройство; для NPU использовать **torch_npu**.
  - **Визуализация**: `visualization_output_dir` — каталог для сохранения графиков (если пусто — `output_dir/visualization`); `visualization_enabled` — включать ли построение графиков при запуске пайплайна; `visualization_formats` — форматы файлов (например, `png`, `svg`).
  - **Прогресс**: `progress_file` — писать ли `progress.json` в output_dir; `progress_log_every_n_steps` — логировать шаг раз в N (если не задано — только при сохранении attention row).
- Загрузка и валидация конфига — через pydantic/pydantic-settings; невалидный конфиг должен приводить к явной ошибке с понятным сообщением.

## 5. Устройство вычислений (GPU / CPU / NPU)

- Поддержка **конфигурируемого устройства**: возможность задавать в конфиге устройство для загрузки модели и выполнения forward/generate (`device`).
- Допустимые значения: `"cpu"`, `"cuda"` (или `"cuda:0"`, `"cuda:1"` и т.д.), `"npu"` (или `"npu:0"` и т.д.). При отсутствии значения в конфиге — автоопределение (cuda при наличии, иначе cpu).
- Модель и все входные/промежуточные тензоры, участвующие в inference, должны находиться на выбранном устройстве. Код выбора устройства вынести в переиспользуемую функцию (по строке из конфига возвращать объект device или строку для `.to(device)`); при недоступности запрошенного устройства — явная ошибка с понятным сообщением.
- **NPU**: в качестве API использовать **torch_npu**. Поддержка NPU опциональна (зависит от установки torch_npu и драйверов); при запросе `device: "npu"` и отсутствии torch_npu — сообщение об ошибке с подсказкой по установке torch_npu.

## 6. Датасет

- Формат: JSON, массив объектов. Каждый элемент содержит как минимум поля `question`, `answer`, `split`; допускаются дополнительные поля. Схема при чтении — гибкая (опциональные поля), но типизированная (например, TypedDict или pydantic BaseModel).

---

## 7. Сбор attention

- Использовать `output_attentions=True` при вызове модели (generate или пошаговый forward).
- Сохранять только **текущую строку** матрицы attention для каждого шага: для тензора формы `(batch, num_heads, seq_len, seq_len)` брать `attn[:, :, -1, :]` — форма `(batch, num_heads, seq_len)`.
- Сохранять для всех слоёв и всех голов; не сохранять полную матрицу `seq_len x seq_len` для decode-шагов (для prefill — отдельно по флагу, см. ниже).

---

## 8. Важные токены и дельты

- **Важность**: по текущей строке attention (на усреднённом по головам/слоям или по выбранной агрегации) накопить веса по позициям, отсортировать по убыванию, взять минимальное число позиций с суммой ≥ `importance_threshold` (0.95). Эти позиции — «важные» на данном шаге.
- **Число важных токенов**: сохранять **без нормализации** — целое число (сырой счётчик), без деления на seq_len.
- **Дельты на каждом шаге**:
  - `newly_important` — токены, вошедшие в текущее множество важных и не входившие на предыдущем шаге;
  - `no_longer_important` — токены, входившие на предыдущем шаге и не входящие в текущее.
  - На первом шаге: только `newly_important`; `no_longer_important` пусто/ноль.
- **Дельты по слоям**: на каждом шаге для каждого слоя множество важных позиций — **объединение** важных позиций по всем головам слоя (важность головы — по порогу 0.95 по текущей строке этой головы). Для слоя: `newly_important` — токены, вошедшие в это объединение на текущем шаге и не входившие на предыдущем; `no_longer_important` — входившие на предыдущем шаге и не входящие в объединение на текущем (т.е. неважные для всех голов слоя). В метаданных на каждый шаг сохранять для каждого слоя количество newly_important и no_longer_important (сырые счётчики): `newly_important_per_layer`, `no_longer_important_per_layer` (списки длины num_layers).
- **Дельты по головам**: для каждой пары (слой, голова) на каждом шаге считать, сколько токенов стали важными для этой головы (newly_important) и сколько перестали быть важными (no_longer_important), сравнивая с предыдущим шагом. В метаданных на каждый шаг сохранять: `newly_important_per_layer_head`, `no_longer_important_per_layer_head` — списки списков целых чисел `[layer][head]` (сырые счётчики).

---

## 9. Разреженность (sparsity)

- На каждом шаге для **текущего токена** считать разреженность по **каждой голове каждого слоя** по текущей строке attention (вектор длины seq_len).
- Метрика: например, число позиций с весом выше `sparsity_threshold`, или другая однозначно заданная метрика; значения **без нормализации** (сырые счётчики или ненормализованные величины).
- Собирать и сохранять разреженность в том же формате и в те же моменты, что и число важных токенов: по шагам, с разбивкой по слоям и головам.
  - **По головам**: в метаданных на каждый шаг — `sparsity[layer][head]` для всех слоёв и голов (сырые счётчики по каждой голове).
  - **По слоям**: дополнительно сохранять `sparsity_per_layer` — список длины num_layers, для каждого слоя — сумма счётчиков по всем головам этого слоя (сырой счётчик по слою).
- В метаданных на каждый сохраняемый шаг также: `num_important_tokens` (raw).
- Дополнительно на каждом сохраняемом шаге считать и сохранять **sparsity как долю неважных токенов**: `(seq_len - num_important) / seq_len`; в метаданных per_step хранить `seq_len` и `sparsity_proportion`.

---

## 10. Условие сохранения текущей строки

- Сохранять **текущую строку attention** (decode) и запись в **attention_rows/** только когда выполняется хотя бы одно из условий:
  - прошло `save_every_n_steps` шагов с последнего сохранения;
  - число `newly_important` на этом шаге больше `save_when_new_important_above_k`.
- При сохранении записывать: шаг, идентификатор запроса, по слоям/головам — массив весов текущей строки.
- **Метаданные** записываются на каждый шаг: в память (per_step) и **файл `metadata.json` перезаписывается после каждого шага декодирования** (актуальный снимок для отладки и при прерывании пайплайна). В метаданных — дельты (глобальные и по слоям), число важных токенов без нормализации, разреженность по всем головам, флаги появления think-токенов.

---

## 11. Токены размышлений

- Отслеживать по сгенерированному тексту (или по токенам) появление подстрок из `thinking_markers` (в т.ч. `\think` и маркеры вроде "Wait, ...").
- Запоминать шаг (и при необходимости позицию) первого появления каждого маркера.
- В метаданных сохранять список событий вида `{ "marker": "...", "step": k }` и т.п.

---

## 12. Prefill

- При `save_prefill_attention: true` сохранять скоры attention фазы prefill **отдельно** от decode (отдельная подпапка, например `prefill/` в папке запроса).
- Формат prefill: по слоям/головам матрица внимания формы `(seq_len, seq_len)` в том же ключевом формате, что и decode (единая схема имён), чтобы не смешивать с шагами генерации.

---

## 13. Формат сохранения и читаемость

- Единая схема имён: ключи вида `layer_<L>_head_<H>` или явная индексация по слою/голове во всех артефактах.
- Формат файлов: NPZ с чёткими ключами или JSON с числовыми массивами; при необходимости — `manifest.json` или `format_spec.json` с описанием размерностей и ключей для валидации при чтении.
- Папка на запрос: `output_dir / request_<id>/`:
  - `metadata.json` — обновляется **после каждого шага** декодирования; шаги, дельты, порог 0.95, thinking-события, параметры N/K, флаг prefill; в каждом элементе `per_step` — одна запись на каждый шаг декодирования: num_important_tokens (raw), newly_important_count, no_longer_important_count, опционально `newly_important_per_layer`, `no_longer_important_per_layer` (списки длины num_layers), опционально `newly_important_per_layer_head`, `no_longer_important_per_layer_head` (списки списков [layer][head]), sparsity по слоям/головам (`sparsity[layer][head]`), `sparsity_per_layer` (список длины num_layers — сумма по головам слоя), seq_len, sparsity_proportion.
  - `attention_rows/` — текущие строки attention (decode) только для сохранённых шагов (например, `step_<k>.npz`).
  - При включённом prefill — `prefill/` с матрицами по слоям/головам.
- Общие артефакты: сгенерированные ответы (JSON/JSONL с request_id, prompt, generated_text, steps и т.д.); копия или путь к датасету (например, `output_dir/dataset_used.json`).
- **Трекинг прогресса**: опциональный файл в `output_dir` (например, `progress.json`) с полями текущего запроса (current_request_index, total_requests, request_id) и текущего шага (current_step, max_output_len). Обновляется после каждого шага; атомарная запись.

## 14. Чтение и парсинг выходов

- Отдельный модуль (например, `read_outputs.py`) с API для загрузки результатов.
- Функции: загрузка `metadata.json` в типизированную структуру (dataclass/pydantic); загрузка decode attention по запросу и шагу (по слоям/головам); загрузка prefill attention при наличии.
- Возвращаемые структуры должны быть пригодны для анализа (numpy-массивы, списки шагов, дельты, thinking-события, per-step sparsity и num_important_tokens). При необходимости проверять версию формата или format_spec для совместимости.

## 15. Визуализация

- **Назначение**: постобработка готовых выводов пайплайна без вызова модели. Чтение данных из `output_dir` (metadata.json, при необходимости attention_rows) через API read_outputs; построение графиков и сохранение в каталог и форматы, заданные в конфиге.
- **Входные данные**: `metadata.json` по каждому запросу (per_step: шаги, num_important_tokens, newly_important_count, no_longer_important_count, sparsity, sparsity_per_layer, newly_important_per_layer_head, no_longer_important_per_layer_head, seq_len, sparsity_proportion); при построении распределения скоров — сохранённые attention rows (step_<k>.npz).
- **Результаты**: графики динамики (число важных токенов по шагам; динамика добавления новых важных токенов и удаления старых — newly/no_longer; динамика доли неважных токенов — sparsity proportion по шагам); общее распределение скоров (гистограмма весов attention по сохранённым шагам); статистика разреженности (по per_step sparsity — по шагам, heatmap по слоям/головам, sparsity proportion по шагам).
- **Сохранение**: в каталог `visualization_output_dir` (или `output_dir/visualization`), форматы из `visualization_formats`. Имена файлов вида `request_<id>_importance_dynamics`, `request_<id>_sparsity_stats`, `request_<id>_score_distribution`, а также сводный `score_distribution_all`.
- **Запуск**: из пайплайна после записи ответов (если `visualization_enabled`) или отдельной командой: `python -m visualization.generate <config>`.

## 16. Тесты

- Тесты только на **pytest** (без unittest). Все тесты в каталоге `tests/`, с типизацией и docstrings.
- Для логики расчётов (важность, дельты, разреженность, условие сохранения) использовать **искусственные данные** — без вызова реальной модели.
- Покрыть: расчёт важности по порогу 0.95; подсчёт дельт (newly_important, no_longer_important) в граничных случаях; **расчёт дельт по слоям** (union по головам, граничные случаи); **расчёт дельт по головам** (newly_important_per_layer_head, no_longer_important_per_layer_head); условие сохранения по N и K; разреженность по слоям и по головам (sparsity_per_layer, sparsity[layer][head]) и число важных без нормализации; запись/чтение метаданных и attention-блоков в темповую папку; проверка, что в metadata per_step содержится запись на каждый шаг и что **файл metadata.json перезаписывается после каждого шага**; чтение полей `newly_important_per_layer`, `no_longer_important_per_layer`, `sparsity_per_layer`, `newly_important_per_layer_head`, `no_longer_important_per_layer_head`; при наличии — **трекинг прогресса** (progress.json); парсинг выходов через read_outputs (запись искусственных артефактов и проверка чтения); **визуализацию** — обнаружение запросов и агрегация метрик, построение графиков на искусственных данных, интеграционный прогон generate с темповым output_dir.
- Импорты в тестах: при необходимости использовать `TYPE_CHECKING` для `CaptureFixture`, `FixtureRequest`, `LogCaptureFixture`, `MonkeyPatch`, `MockerFixture` и т.п.
